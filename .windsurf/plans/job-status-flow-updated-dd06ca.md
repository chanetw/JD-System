# 📋 ปรับปรุงสถานะงาน DJ System — แผนปรับปรุง (Update)

แผนนี้ปรับปรุงการแสดงสถานะงานในหน้า DJ List ให้ชัดเจน โดยแยกระหว่าง "สถานะอนุมัติ" และ "สถานะงาน" และเพิ่มระบบแจ้งเตือนสำหรับงานที่ไม่มีผู้รับงาน

---

## 🎯 ปัญหาปัจจุบัน

1. หน้า DJ List แสดงสถานะ `approved` และ `in_progress` เหมือนกัน ทำให้สับสน
2. ไม่มีการแยกระหว่างสถานะการอนุมัติ (Approval Flow) และสถานะการทำงาน (Work Flow)
3. การปฏิเสธจาก Approver และ Assignee ดูเหมือนกัน
4. ไม่มีการแจ้งเตือนเมื่องานอนุมัติแล้วแต่ไม่มีผู้รับงาน
5. สถานะ `pending_close` มี UI แต่ไม่ได้ใช้งาน
6. **ระบบ In-App Notification ปัจจุบันไม่แสดงผลเลยสำหรับทุก Role**

---

## ✅ แผนการแสดงสถานะใน DJ List (ใหม่)

ตารางนี้คือ Mapping ที่ถูกต้องสำหรับแสดงในหน้า DJ List โดยแยกเป็น 2 Column ชัดเจน

| สถานะในระบบ (Backend) | **สถานะอนุมัติ (Approval Status)** | **สถานะงาน (Work Status)** | อธิบาย |
|-----------------------|------------------------------------|----------------------------|---------|
| `pending_approval` | รออนุมัติ 🟡 | - | รอ Approver พิจารณา |
| `approved` | อนุมัติแล้ว 🟢 | **ยังไม่มอบหมาย** 🟡 | อนุมัติแล้ว แต่ยังไม่มี Assignee (รอ Admin มอบหมาย) |
| `assigned` | อนุมัติแล้ว 🟢 | **ได้รับมอบหมาย** 🔵 | มี Assignee แล้ว แต่ยังไม่เริ่มเปิดดู |
| `in_progress` | อนุมัติแล้ว 🟢 | **กำลังดำเนินการ** 🟣 | Assignee เปิดดูและเริ่มทำงาน (เริ่มนับ SLA) |
| `completed` | อนุมัติแล้ว 🟢 | **เสร็จสมบูรณ์** 🟢 | Assignee ส่งมอบงานแล้ว |
| `rejected` | **ไม่อนุมัติ** 🔴 | - | Approver ปฏิเสธในขั้นตอนการอนุมัติ |
| `rejected_by_assignee`| อนุมัติแล้ว 🟢 | **ถูกปฏิเสธ** 🟠 | Assignee ปฏิเสธการทำงาน (หลังจากได้รับมอบหมาย) |

## 🔄 Flow ที่ถูกต้องพร้อม Notification

```
Requester สร้างงาน
    │
    ▼
[pending_approval] ← รอ Approver อนุมัติ
    │
    ▼ Approver อนุมัติ
[approved] ← อนุมัติแล้ว แต่ยังไม่มี assignee
    │
    ├── 📢 **แจ้งเตือน Admin**: "มีงานอนุมัติแล้ว รอมอบหมาย"
    │
    ├── ถ้ามี Auto-Assign
    │       │
    │       ▼
    │   [assigned] ← มอบหมายอัตโนมัติ
    │
    └── ถ้าไม่มี Auto-Assign
            │
            ▼ Admin มอบหมาย (Manual Assign)
        [assigned] ← มี assignee แล้ว
            │
            ▼ Assignee เปิดดู (Auto-start)
        [in_progress] ← เริ่มทำจริง
```

## 🔔 ระบบ Notification ที่ต้องเพิ่ม

| Event | แจ้งใคร | ช่องทาง | เนื้อหา |
|-------|----------|---------|---------|
| งานอนุมัติแล้ว + ไม่มี assignee | **Admin ทั้งหมด** | In-App + Email | "มีงาน #{djId} อนุมัติแล้ว รอการมอบหมาย" |
| งานอนุมัติแล้ว + มี Auto-Assign | Assignee | In-App | "ได้รับมอบหมายงาน #{djId}" |
| งานรออนุมัติ | Approver | In-App + Email | "มีงาน #{djId} รอการอนุมัติ" |

## 📋 สิ่งที่ต้องทำ

### 1. ปรับ UI หน้า DJ List
- **เพิ่ม/ปรับ Column**: แยกระหว่าง "สถานะอนุมัติ" และ "สถานะงาน" ตามตาราง Mapping
- **ปรับสี Badge**: ให้สอดคล้องกับความหมายของแต่ละสถานะ (เช่น สีแดง=ไม่อนุมัติ, สีส้ม=ถูกปฏิเสธ)

### 2. ตรวจสอบและซ่อมแซมระบบ In-App Notification
- **ปัญหา**: ปัจจุบันไม่มีการแสดงผล In-App Notification สำหรับทุก Role
- **สิ่งที่ต้องทำ**:
  - ตรวจสอบ `Notification` model ใน Prisma
  - ตรวจสอบ API route สำหรับดึง/สร้าง Notification
  - ตรวจสอบ UI Component (เช่น รูปกระดิ่งบน Navbar) ว่ามีการ fetch ข้อมูลและแสดงผลถูกต้องหรือไม่
  - ตรวจสอบ WebSocket/SSE (ถ้ามี) หรือ Polling สำหรับ Real-time update

### 3. เพิ่ม เหตุการณ์ (Events) ในระบบ Notification
- **Backend**: สร้าง notification service ให้ครอบคลุม:
  - Job `approved` + ไม่มี `assigneeId` (แจ้ง Admin รอมอบหมาย)
  - Job `approved` + มี `assigneeId` (แจ้ง Assignee ว่าได้รับมอบหมายแบบ Auto-Assign)
  - Job `pending_approval` (แจ้ง Approver ว่ามีงานรออนุมัติ)

### 4. จัดการสถานะอื่นๆ
- **JobActionPanel**: ปรับเงื่อนไขการแสดงปุ่มให้สอดคล้องกับสถานะใหม่
- **MyQueue (Assignee)**: จัดหมวดหมู่ Tab ให้ตรงกับสถานะงาน
- **ปิดใช้งาน `pending_close`**: ซ่อน UI และให้ส่งงานเปลี่ยนเป็น `completed` ทันที

## 🎯 ผลที่ได้

- **ลดความสับสน**: ผู้ใช้เห็นชัดเจนว่างานติดอยู่ที่ขั้นตอนการอนุมัติ หรือขั้นตอนการทำงาน
- **แยกการปฏิเสธ**: ชัดเจนว่า "ไม่อนุมัติ" (Approver) หรือ "ถูกปฏิเสธ" (Assignee)
- **ไม่ตกหล่น**: Admin ได้รับแจ้งเตือนเมื่อมีงานรอมอบหมาย
- **ประสิทธิภาพ**: Assignee รู้ทันทีเมื่อได้รับมอบหมาย
