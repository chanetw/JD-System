# สรุปการแก้ไข — 26 มีนาคม 2026

## รายการแก้ไข

### 1. Admin Page Refresh → 502 Nginx
- **อาการ:** หน้า Admin refresh แล้วบางครั้งขึ้น 502 Bad Gateway (nginx HTML page)
- **สาเหตุ:** Backend container ยังไม่พร้อม หรือ Docker DNS resolve ล่าช้า → nginx proxy ไป `backend:3000` ไม่ได้
- **แก้ไข:**
  - ลด resolver TTL จาก `30s` → `10s` เพื่อให้ re-resolve DNS เร็วขึ้น
  - เพิ่ม `proxy_connect_timeout 10s`
  - เพิ่ม `proxy_intercept_errors on` + `error_page 502 = @backend_down`
  - สร้าง `@backend_down` location ส่ง JSON 503 แทน nginx HTML page
  - เพิ่ม `error_page 502 503 504 /50x-fallback.html` สำหรับ non-API requests (fallback ไป SPA index.html)
- **ไฟล์:** `frontend/nginx.conf`
- **ผลกระทบ DB:** ❌ ไม่มี

### 2. สมัครสมาชิก → behavior ของ Default HO (REVERTED)
- **อาการ (ก่อน revert):** สมัครสมาชิกใหม่จะสำเร็จแม้ระบบจะไม่มี project ที่มี `code = 'HO'` — ผู้ใช้ถูกสร้างโดยไม่มี default project scope แต่ได้รับข้อความแยกต่างหากให้ติดต่อผู้ดูแล
- **การตัดสินใจ:** คืนพฤติกรรมเดิมเพื่อบังคับให้ระบบมี Default HO project อยู่ — การเปลี่ยนแปลงนี้ revert แล้ว
- **รายละเอียดการ revert:**
  - คืนการ throw ข้อผิดพลาด `DEFAULT_PROJECT_NOT_FOUND` เมื่อไม่พบ default HO project
  - คืนการ assign default scope ให้เป็นแบบบังคับ (หากไม่พบจะไม่สร้างผู้ใช้)
  - ปรับข้อความ success ใน `v2/index.js` กลับเป็นข้อความที่ระบุว่า user ได้รับ Default Requester access (HO project) เมื่อสำเร็จ
- **ไฟล์ที่ถูก revert / ที่เกี่ยวข้อง:**
  - `backend/api-server/src/v2/adapters/PrismaV1Adapter.js` — คืน `throw new Error('DEFAULT_PROJECT_NOT_FOUND')` และบังคับการสร้าง `userScopeAssignment`
  - `backend/api-server/src/v2/index.js` — คืน success message แบบเดิม
- **ผลกระทบ DB:** ❌ ไม่มี schema change แต่: ถ้าฐานข้อมูล tenant ไม่มี project ที่มี `code = 'HO'` การสมัครสมาชิกจะล้ม (ต้องสร้าง record `HO` ก่อน)
- **คำแนะนำปฏิบัติการ:** ตรวจสอบฐานข้อมูล production ว่ามี project `HO` สำหรับ tenant เป้าหมายหรือไม่ หากไม่มีย้อนกลับให้สร้าง project `HO` ก่อนเปิดรับสมัคร

### 3. User Management → Reset Password → 404
- **อาการ:** กด Reset Password ในหน้า User Management → ขึ้น 404 Request Failed
- **สาเหตุ:** Frontend เรียก `PUT /api/users/{id}/reset-password` แต่ endpoint จริงอยู่ที่ `PUT /api/v2/users/{id}/reset-password`
- **แก้ไข:** เปลี่ยน path ใน `adminService.js` จาก `/users/${userId}/reset-password` → `/v2/users/${userId}/reset-password`
- **ไฟล์:** `frontend/src/modules/shared/services/modules/adminService.js` (line 1465)
- **ผลกระทบ DB:** ❌ ไม่มี

### 6. ลบ Job Type ไม่ได้ / ลบแล้วกลับมา (Soft Delete vs Hard Delete)
- **อาการ:** กดลบ Job Type แล้วบางรายการดูเหมือนลบไม่สำเร็จ หรือกลับมาแสดงอีก
- **สาเหตุ:** เดิมเป็น soft delete อย่างเดียว และหน้า admin ยังแสดงรายการ inactive บางกรณี
- **แก้ไข:**
  - เพิ่ม logic ใน backend: ถ้าลบแล้ว **ไม่พบ dependency** ให้ลบถาวร (hard delete)
  - ถ้ามี dependency (jobs / assignments / approval flows / chain) ให้ทำ soft delete (`isActive=false`)
  - เพิ่ม tenant access check ก่อนแก้ไข/ลบ job type
  - เพิ่ม `status` ใน list response ให้สอดคล้องกับ `isActive`
  - ปรับหน้า admin ให้ซ่อนรายการ soft-deleted จากหน้าจัดการ
  - ปรับการ toggle สถานะใน UI ให้ใช้ `isActive` เป็น source of truth
- **ไฟล์:**
  - `backend/api-server/src/routes/job-types.js`
  - `frontend/src/modules/features/admin/pages/JobTypeSLA.jsx`
- **ผลกระทบ DB:** ⚠️ มีเฉพาะกรณี hard delete (เฉพาะรายการที่ไม่ถูกอ้างอิง), ไม่มี schema change / migration
